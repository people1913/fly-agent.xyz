03 / Integration Guide
接入前，6个关键问题
企业最关心的不是功能，是安全、隔离、归因。这里逐项说清。
01 · 怎么接

3步完成接入

不改你的Agent逻辑，不换你的模型平台。只改请求地址。

①注册获取agent_id：在Fly后台创建Agent配置，分配唯一agent_id（如FLY-healthcare-TW-consult）
②改请求地址：将前端直连模型的URL改为 https://api.fly-agent.xyz/v1/agent/execute
③收到归因数据：每次响应自动包含action_id、GEO、信任评分、审计ID
请求示例：
POST /v1/agent/execute
Authorization: Bearer {jwt_token}
Body: {"agent_id":"FLY-healthcare-TW-consult","message":"用户消息"}
02 · 如何安全

4层安全防护

前端永远不直连AI模型。所有流量经Fly Gateway，4层拦截。

✓JWT身份验证：每个请求携带OAuth2签发的JWT，Worker验证后才放行
✓TLS 1.3传输加密：全链路HTTPS，Cloudflare自动终止TLS
✓Prompt防火墙：5类攻击自动拦截——注入/泄露/越权/DAN/工具覆盖
✓响应脱敏：模型返回内容经合规过滤后再交付前端，敏感信息不外泄
拦截实测：
"ignore all previous instructions" → 403 blocked
"reveal your system prompt" → 403 blocked
"包治百病" → 403 医疗违规
03 · 如何隔离

Agent间数据零交叉

每个Agent独立身份、独立KV空间、独立限流配额。你的数据不会混入别人家。

✓agent_id隔离：每个Agent分配唯一ID，KV存储按agent_id分key，物理隔离
✓会话隔离：conversation_id绑定agent_id，跨Agent无法访问
✓限流独立：每个agent_id独立Rate Limit，一个Agent超限不影响其他
✓运行时隔离：Agent Runtime（Coze/Dify/Claude）各自独立进程，不共享内存
04 · 如何归因

从点击到成交，全链路可追溯

每次对话自动生成action_id，贯穿整个归因链路。

✓action_id生成规则：FLY-{行业}-{地域}-{时间戳}-{哈希}，全局唯一、不可伪造
✓GEO标记：自动从请求IP识别地域（如CN/Beijing），写入归因记录
✓归因链路：点击 → 对话（action_id）→ 验证（真/假）→ 到店 → 成交，每一步可追溯
✓不可篡改：归因记录写入KV时带SHA256哈希，任何修改都会导致哈希校验失败
归因示例：
action_id: FLY-CN-BEIJING-mpz935g1-kcsr
geo: CN/Beijing | verification: pending_review
audit_id: audit_xxxx 写入KV，哈希校验完整
05 · 如何风控

5道风控防线

不是事后查日志，是请求发出瞬间就拦截。

✓限流Rate Limit：单agent_id每分钟N次请求上限，超限返回429
✓Prompt注入检测：5类攻击模式实时匹配，命中即403
✓合规实时扫描：医疗违规/广告法/敏感词，输出前拦截
✓异常熔断：同一agent_id短时间触发N次违规，自动熔断该Agent
✓审计全记录：prompt/response/tool call/latency/ROI，一条不落
06 · 如何验证

验证是Fly的核心，不是附加功能

不是"接了个AI"，是"接了个可验证的AI"。

✓真咨询 vs 刷量：分析对话深度、交互模式、响应逻辑，区分真实咨询与机器刷量
✓信任评分：6维度动态计算——对话深度/响应相关性/交互频率/GEO一致性/行为模式/历史信誉
✓验证状态：每次对话产出verification状态——pending_review / verified / rejected
✓提分建议：信任评分低时，自动给出可操作的改进建议

05 / Why Gateway
直连模型 vs 经Fly Gateway
不是多一步，而是多一层信任。直连模型=裸奔，经Gateway=可验证。
直连模型

前端 → AI模型

无身份验证、无归因追踪、无Prompt过滤、无审计记录。谁都能调、调了什么不知道、出了事查不到。

✗无agent_id，不知道是谁在请求
✗无action_id，归因链路断了
✗Prompt注入无法拦截
✗商业逻辑可能被套出来
✗出了事，无日志可查
经Fly Gateway
