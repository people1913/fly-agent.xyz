## 2. 错误率监控方案

**架构：双层检测**

```
请求流入 → Worker fetch handler
                ↓
         recordRequestMetric() → KV metrics:m:{minute}
                ↓
         checkAlertConditions() → 实时5xx率检测（每请求）
                ↓ (5min后)
         scheduled handler → getAggregatedMetrics(5) → 慢查询检测
                ↓ (6h后)
         GA trend-check → CF GraphQL → 全局错误率检测
```

**第一层：Worker实时检测（已有）**
- 每个请求结束后异步调用 `checkAlertConditions()`
- 聚合最近5分钟KV数据
- 5xx率>10%=P0, >3%=P1
- P95延迟>3000ms=P1

**第二层：GA CF Analytics（已有）**
- 每6小时通过CF GraphQL查询全局错误率
- 补充Worker无法自检的场景（Worker本身crash不写KV）

**需要新增：**
- P99延迟监控（当前只有P95）
- 4xx率监控（当前只监控5xx）
- 请求量突降检测（total < 阈值）

## 8. Telegram通知方案

**当前实现：**
- Worker: `sendTelegramAlert()` → `api.telegram.org/bot{TOKEN}/sendMessage`
- GA: curl直接调用Telegram Bot API
- 格式：emoji + 标题 + 详情 + 时间戳

**告警格式：**
```
🔴/🟡/ℹ️ [P0/P1/INFO] 标题

详情字段...

⏰ 2026-06-11T10:00:00Z
```

**去重机制：**
- KV `alert:{key}` → TTL 30分钟（ALERT_DEDUP_TTL）
- 同类告警30分钟内不重复发送

**无需改动，当前方案已满足。**

## 10. 免费方案优先

所有组件均在免费额度内：

| 组件 | 免费额度 | 预计用量 | 安全边际 |
|------|----------|----------|----------|
| CF Workers | 100k req/day | ~1000/day | 99% |
| CF KV | 100k read/day, 1k write/day | ~5k read, ~200 write | 95%/80% |
| CF D1 | 5GB, 5M read/day | <1MB, <1k read/day | >99% |
| CF R2 | 10GB | 0 (未用) | 100% |
| GitHub Actions | 2000 min/month | ~300 min/month | 85% |
| Telegram Bot | 免费 | <100 msg/day | >99% |
| Resend | 100 emails/day | <10/day | 90% |

**唯一风险：KV writes**
- 当前：metrics bucket每分钟1次write + scheduled每5分钟写探针 + 告警去重
- 预计：~500 writes/day，占免费额度50%
- 缓解：合并metrics写入间隔（1分钟→5分钟），告警去重TTL从30分钟延长到1小时

## 验收标准

每个模块必须同时提供5项运行证据：

| 模块 | GitHub存在 | Worker存在 | API存在 | 数据存在 | 端到端存在 |
|------|-----------|-----------|---------|---------|-----------|
| Health Check | workflow文件SHA | /v1/health返回ok | curl响应200 | db=ok, kv=ok | GA运行conclusion=success |
| 错误率监控 | workflow文件 | recordMetric写入KV | /v1/admin/metrics返回数据 | KV有metrics bucket | 5xx触发P0告警+收到Telegram |
| D1监控 | workflow文件 | scheduled D1探针 | D1 SELECT 1成功 | daily_metrics有行 | 探针失败→P0告警 |
| KV监控 | workflow文件 | scheduled KV探针 | put→get一致 | capacity:kv有值 | 探针失败→P0告警 |
| 备份监控 | workflow文件 | POST /backup成功 | 返回backup_id | 213+ records | 失败→P0告警 |
| GA失败监控 | workflow文件 | notify-failure job | if: always() | 收到Telegram通知 | 失败→双通道通知 |
| 查询端点 | (新) | Worker代码 | /v1/admin/status | 返回完整状态 | curl可查 |

**禁止：**
- ❌ "代码写了所以应该可以"
- ❌ "部署了所以已完成"
- ❌ "设计文档里写了"
- ✅ 只接受：curl响应 + GA运行日志 + D1记录 + Telegram截图 + 邮件截图
