Attribution

Verifiable Attribution Model
核心定义
Attribution = 可验证的归因
Fly 的归因不是"猜测谁贡献了转化"，而是"验证这个转化是否真实发生、来源是否可信、链路是否完整"。
🔑
Fly 不是渠道，Fly 验证渠道。抖音、小红书、微信、美团、飞书是信号来源，Fly 是对这些信号进行验证、归因、审计的基础设施。Fly 不产生信号，Fly 让信号可信。

信号传递机制

Action ID + 哈希链

每个事件发生时：①写入当前事件数据 + Action ID → ②计算当前事件 SHA-256 哈希 → ③将前一个事件的哈希作为输入计算新哈希 → ④追加到哈希链末尾。任何中间环节的数据修改都会导致后续哈希链断裂。

四大约束

单调递增：事件时间戳必须严格递增

阶段有序：推荐→咨询→成交→结算，不能跳过或逆序

哈希连续：每个事件的哈希必须包含前一个事件的哈希

签名验证：每个事件由产生该事件的 Agent 签名

验证机制

GATE 1
来源验证：确认信号来自声称的渠道
GATE 2
因果验证：确认事件之间的因果关系成立
GATE 3
完整性验证：确认哈希链无断裂
GATE 4
时间验证：确认事件时序合理，无回跳或异常间隔

失败场景

场景1：伪造来源

攻击者伪造渠道标识。
拦截：来源验证——校验渠道回调签名和推荐人身份，GATE 1 拦截。

场景2：归因劫持

攻击者在用户成交前替换 Action ID。
拦截：哈希链验证——中间替换导致链断裂。

场景3：刷单作弊

攻击者批量生成虚假咨询和成交。
拦截：因果验证 + 时间验证——自动标记 ANOMALY。

场景4：数据篡改

内部人员修改成交金额或推荐人信息。
拦截：哈希链 + 签名——修改导致哈希不匹配和签名失效。

场景5：重复结算

同一个 Action ID 试图多次触发佣金结算。
拦截：唯一性验证——SETTLED 终态，永久锁定。
