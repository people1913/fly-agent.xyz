Attribution

Verifiable Attribution Model
核心定义
Attribution = 可验证的归因
Fly 的归因不是"猜测谁贡献了转化"，而是"验证这个转化是否真实发生、来源是否可信、链路是否完整"。

渠道归因

什么是渠道归因

渠道归因确定"信号从哪个渠道来"。Fly 不是简单地标记来源，而是验证来源——确认信号确实由该渠道产生，而非伪造或劫持。

支持的归因渠道

抖音 / 小红书 / 微信 / 美团 / 飞书——每个渠道有独立的信号适配层，将不同格式的渠道数据统一转换为 Fly 标准验证格式。

渠道信号类型适配方式验证重点
抖音短视频/直播推荐Callback + 前端埋点防刷单、防虚假互动
小红书笔记/种草推荐前端埋点 + 深链防刷赞、防水军
微信公众号/小程序/私域深度链接 + Callback防裂变作弊
美团到店/团购API对接防虚假订单
飞书企业推荐/社群API + Webhook防内部作弊

事件归因

什么是事件归因

事件归因确定"哪个具体行为触发了转化"。Fly 不是简单记录事件，而是验证事件链的因果关系——确认推荐事件确实导致了咨询，咨询确实导致了成交。

R

推荐事件（Referral）

用户通过推荐入口进入。验证：入口是否合法、推荐人是否有效、是否为真实用户。

C

咨询事件（Consultation）

用户发起咨询/互动。验证：咨询是否由推荐触发、Action ID 链路是否连续。

V

成交事件（Conversion）

用户完成交易。验证：订单是否与 Action ID 绑定、金额是否合理。

S

结算事件（Settlement）

佣金分配。验证：全链路哈希链是否完整、Trust Score 是否达标。

信号传递

信号如何从推荐传递到结算

信号传递的核心机制是 Action ID + 哈希链。每个事件发生时：

① 写入当前事件数据 + Action ID

② 计算当前事件的 SHA-256 哈希

③ 将前一个事件的哈希作为输入，计算新哈希

④ 追加到哈希链末尾

这保证了信号传递的连续性和不可篡改性——任何中间环节的数据修改都会导致后续哈希链断裂。

信号传递的约束

单调递增：事件时间戳必须严格递增，不能回跳

阶段有序：推荐→咨询→成交→结算，不能跳过或逆序

哈希连续：每个事件的哈希必须包含前一个事件的哈希

签名验证：每个事件由产生该事件的 Agent 签名

验证机制

验证 1
来源验证：确认信号来自声称的渠道
验证 2
因果验证：确认事件之间的因果关系成立
验证 3
完整性验证：确认哈希链无断裂
验证 4
时间验证：确认事件时序合理，无回跳或异常间隔

失败场景

场景1：伪造来源

攻击者伪造渠道标识，试图将非推荐流量标记为推荐流量。
拦截：来源验证——系统校验渠道回调签名和推荐人 Agent 身份，伪造信号在 GATE 1 被拒绝。

场景2：归因劫持

攻击者在用户成交前替换 Action ID，试图将成交归因到自己的推荐。
拦截：哈希链验证——Action ID 从推荐到成交的哈希链必须连续，中间替换会导致链断裂。

场景3：刷单作弊

攻击者批量生成虚假咨询和成交，试图骗取佣金。
拦截：因果验证 + 时间验证——虚假事件的时序和因果关系通常不合理（如0秒内完成推荐→成交），系统自动标记 ANOMALY。

场景4：数据篡改

内部人员修改成交金额或推荐人信息。
拦截：哈希链 + 签名——每个事件由产生它的 Agent 签名，修改任何字段都会导致哈希不匹配和签名失效。

场景5：重复结算

同一个 Action ID 试图多次触发佣金结算。
拦截：唯一性验证——Action ID 一旦进入 SETTLED 终态，永久锁定，不可重复使用。

归因数据模型

Action（动作）

归因的基本单元。每个 Action 包含 actionId、source（渠道）、referrer（推荐人）、status（验证状态）、hash（哈希值）。

Verification（验证记录）

对 Action 每个阶段的独立验证。每个 Verification 包含 actionId、stage（阶段）、result（结果）、timestamp（时间戳）。

Audit Event（审计事件）

链式审计记录。每个 Audit Event 包含前一个事件的哈希，形成不可篡改的审计链。
