# Fly/CTRS 商业场景验证：10 大真实 Agent Economy 商业场景

> 证明 Agent Economy 为什么需要 Fly：10 个端到端真实商业场景，用 CTRS v1.2 逐环节验证。

---

## 为什么要做场景验证

CTRS v1.2 规范已完成，但规范完成 ≠ 问题被解决。这份文档的目标不是解释字段定义（那是 specification.md 的工作），而是回答一个根本问题：

**在真实的 Agent Economy 商业场景中，Fly 的信任链路能否走通？**

```
商业行为 → 证据（什么证据？）→ 商业事实（CTRS Report 里有什么？）→ 归因（规则怎么分配？）→ 结算（谁拿多少钱？）
```

每个场景我们问三个问题：
1. **没有 Fly，这个场景的痛点是什么？** — 证明需求真实存在
2. **CTRS v1.2 能否完整支撑？** — 证明规范可用
3. **如果不能，最小扩展是什么？** — 证明路线图清晰

### 核心原则

- 不为了协议而协议——每个场景都要问：能不能让一笔钱更容易结算？
- 10/10 都能走通（不是 6/10），区别只是部分需要更高版本增强
- 客户买的不是 Verify，是"敢拿去分钱"
- **冻结原则**：只有当多个真实场景都遇到同一个限制时，才修改 Specification

## 场景 1：AI 导购分佣

### 场景标题

多 Agent 导购协作分佣：推荐 Agent 引流，购买 Agent 支付，谁来分佣金？

### 参与者

| Agent | 角色 | 所属方 |
|-------|------|--------|
| ShopGuide-AI | 导购推荐 | 导购平台 A |
| CompareBot | 跨平台比价 | 比价平台 B |
| CouponAgent | 优惠券匹配 | 优惠平台 C |
| PayAgent-X | 支付执行 | 支付服务 D |

### 商业行为

用户通过导购 Agent 浏览商品，比价 Agent 确认最优价格，优惠券 Agent 匹配折扣，支付 Agent 完成下单——四个不同方的 Agent 协作促成一笔购买。

### 金额示例

| 项目 | 金额 |
|------|------|
| 商品售价 | $200 |
| 平台佣金率 | 8%（Amazon 通用品类基准） |
| 佣金池 | $16.00 |
| 导购 Agent 分佣 | $6.40（40%） |
| 比价 Agent 分佣 | $4.80（30%） |
| 优惠券 Agent 分佣 | $1.60（10%） |
| 支付 Agent 分佣 | $3.20（20%） |

> 行业参考：Amazon Associates 佣金 1%–45%（品类差异），淘宝客 Cookie 有效期 15 天，Honey 事件暴露了 last-click 劫持佣金的系统性风险。

### 归因问题：为什么需要 Fly？

| 没有Fly | 后果 |
|---------|------|
| 电商平台只看 last-click，优惠券 Agent 在结账前自动替换链接 | 导购和比价的贡献被完全忽略 → Honey 式劫持 |
| 导购对话在平台 A、比价在平台 B、支付在平台 D——三方互不信任 | 无法形成共同认可的商业事实 |
| 跨设备归因断裂（手机浏览、电脑下单） | 归因链丢失，佣金无法分配 |
| 各方数据孤岛，没有防篡改机制 | 作假空间大，商业信任无法建立 |

### 完整链路

```
商业行为                Evidence                    Commercial Fact           Attribution               Settlement
─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
导购Agent推荐降噪耳机  → conversation (T1)         → 用户确实通过 Agent      → Rule: U型归因            → 佣金$16按比例:
                       source: shopguide-ai          链路购买了$200商品         推荐40%/比价30%           导购$6.40
比价Agent确认最优价    → conversation (T2)                                      优惠10%/支付20%           比价$4.80
                       source: comparebot                                                                  优惠$1.60
优惠券Agent匹配折扣    → webhook (T3)                                                                     支付$3.20
                       source: couponagent
支付Agent完成下单      → payment (T4)
                       source: payagent-x
电商平台确认订单       → webhook (T5)
                       source: merchant-platform
```

### CTRS v1.2 验证

| 验证项 | 状态 | 说明 |
|--------|------|------|
| Claim → 多方声明 | ✅ 完全支撑 | 4 个 Agent + 用户均可声明为 party |
| Evidence 覆盖 | ✅ 完全支撑 | `conversation`、`webhook`、`payment` 三种类型覆盖全链路 |
| Rule 一等对象 | ✅ 完全支撑 | U 型归因规则，Hash 绑定防篡改 |
| Attribution 计算 | ✅ 完全支撑 | 4 方贡献百分比 + 归因金额 |
| Settlement 分润 | ✅ 完全支撑 | $16.00 按 40/30/10/20 分配，金额加总正确 |
| VR-101~VR-402 | ✅ 全部通过 | Core 级别 20 条验证规则均满足 |

⚠️ **边缘情况**：归因窗口（推荐到购买可能跨越 7 天），可通过 Rule definition 的 `parameters.attribution_window_days` 扩展，不需要规范变更。

### 一句话结论

这个场景证明 Fly 解决了 **多 Agent 推荐链路中 last-click 劫持佣金** 的问题——没有可验证的信任链，导购 Agent 永远拿不到合理分佣。

## 场景 2：AI 招聘归因

### 场景标题

多 Agent 招聘协作归因：简历筛选 Agent 发现人才，猎头 Agent 推荐，谁该拿猎头费？

### 参与者

| Agent | 角色 | 所属方 |
|-------|------|--------|
| ScreenAgent | 简历筛选 | 招聘 AI 平台 A |
| InterviewBot | AI 面试评估 | 面试平台 B |
| HeadhunterAgent | 猎头关系维护 | 猎头公司 C |
| BackgroundAgent | 背景调查 | 背调服务 D |
| HR-Decider | HR 决策辅助 | 企业内部 |

### 商业行为

AI 筛选 Agent 从 1000 份简历中筛选出 10 人，面试 Agent 评估技能和文化匹配，猎头 Agent 维护候选人关系并谈判薪酬，背调 Agent 验证资质，HR Agent 综合建议——5 个 Agent 协作完成一次高管招聘。

### 金额示例

| 项目 | 金额 |
|------|------|
| 候选人年薪 | $30,000 |
| 猎头费率 | 25%（行业基准 20%–35%） |
| 总猎头费 | $7,500 |
| 简历筛选 Agent | $1,125（15%） |
| 面试评估 Agent | $1,500（20%） |
| 猎头 Agent | $3,375（45%） |
| 背调 Agent | $750（10%） |
| HR 决策 Agent | $750（10%） |

> 行业参考：猎头费率通常为年薪 20%–35%；"3331"付款模式（签约30%、推荐30%、入职30%、过保10%）；同一候选人被多家猎头推荐时需支付"重复推荐费"。

### 归因问题：为什么需要 Fly？

| 没有Fly | 后果 |
|---------|------|
| 候选人在 LinkedIn 被 AI 发现、在招聘网站投简历、被猎头推荐——三个渠道都声称"首次引入" | 来源追溯困难，猎头独占费用 |
| AI 筛选了 90% 的初始工作但无独立收费机制 | AI 贡献不可见，成本结构失衡 |
| 候选人入职 2 个月后离职，猎头免费补招，但 AI 工具不被重新补偿 | 保证期争议无法量化 |
| 多猎头撞单，企业需支付重复推荐费 | 无法验证"谁先发现" |

### 完整链路

```
商业行为                Evidence                    Commercial Fact           Attribution               Settlement
─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
筛选Agent发现候选人    → crm (T1)                  → 候选人年薪$30,000       → Rule: 任务权重归因       → 猎头费$7,500按比例:
                       source: screenagent           已确认入职                 筛选15%/面试20%           筛选$1,125
面试Agent评估能力      → conversation (T3)                                      猎头45%/背调10%           面试$1,500
                       source: interviewbot                                    HR决策10%                 猎头$3,375
猎头Agent推荐并谈判    → crm (T5)                                                                         背调$750
                       source: headhunteragent                                                            HR$750
背调Agent验证资质      → custom (T7)
                       source: backgroundagent
HR确认录用发Offer     → webhook (T10)
                       source: hr-decider
```

### CTRS v1.2 验证

| 验证项 | 状态 | 说明 |
|--------|------|------|
| Claim → 多方声明 | ✅ 完全支撑 | 5 个 Agent 作为 party |
| Evidence 覆盖 | ✅ 完全支撑 | `crm`、`conversation`、`custom`、`webhook` 四种类型 |
| Rule 一等对象 | ✅ 完全支撑 | 任务权重归因规则，支持"3331"付款节奏通过 parameters 表达 |
| Attribution 计算 | ✅ 完全支撑 | 5 方贡献百分比 + 归因金额 |
| Settlement 分润 | ✅ 完全支撑 | $7,500 按 15/20/45/10/10 分配 |
| VR-101~VR-402 | ✅ 全部通过 | |

⚠️ **边缘情况**：保证期内候选人离职需重新招聘，当前 Report 是一次性快照，无法自动触发"退款/补招"流程。可通过 Enterprise Extension 的多 Settlement 支持（分期结算中最后一期延迟到过保后）解决。

### 一句话结论

这个场景证明 Fly 解决了 **招聘中 AI 工具贡献不可见** 的问题——没有信任链，AI 筛选 Agent 做了 90% 的工作却拿不到一分钱。

## 场景 3：AI 客服贡献

### 场景标题

客服 Agent 链路归因：技术支持 Agent 解决问题 → 客户续费，客服贡献如何量化？

### 参与者

| Agent | 角色 | 所属方 |
|-------|------|--------|
| TriageAgent | 问题分流 | 客服平台 A |
| TechSupportAgent | 技术问题解决 | 技术支持 B |
| RetentionAgent | 流失预防 | 客户成功 C |
| SuccessAgent | 增购发现 | 客户成功 C |

### 商业行为

客户遇到技术问题，分流 Agent 识别问题类型并路由，技术支持 Agent 解决关键问题，挽留 Agent 识别流失风险并主动干预，客户成功 Agent 发现增购机会——最终客户不仅续费还增购了模块。

### 金额示例

| 项目 | 金额 |
|------|------|
| 客户年订阅 | $12,000 |
| 续费增量（客服贡献） | $600（续费率提升 5%） |
| 增购模块收入 | $3,000 |
| 可归因收入池 | $3,600 |
| 分流 Agent | $360（10%） |
| 技术支持 Agent | $1,440（40%） |
| 挽留 Agent | $1,080（30%） |
| 客户成功 Agent | $720（20%） |

> 行业参考：Klarna AI 客服处理 230 万次对话，利润贡献约 $4,000 万；73% 的公司无法量化客户成功工作的财务影响；NRR 健康值 100%–130%。

### 归因问题：为什么需要 Fly？

| 没有Fly | 后果 |
|---------|------|
| 客服被视为成本中心而非利润贡献者 | 预算永远被压缩，无增量激励 |
| 续费归因模糊——产品好？价格低？还是客服解决了关键问题？ | 客服贡献与收入脱节 |
| 流失预防的"反事实"问题：挽留 Agent 阻止了可能不会流失的客户 | "阻止"的价值难以量化 |
| 增购归属争议——客户成功 Agent 发现机会，销售团队认为是自己谈下来的 | 价值错配，激励冲突 |

### 完整链路

```
商业行为                Evidence                    Commercial Fact           Attribution               Settlement
─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
分流Agent识别问题类型  → conversation (T1)         → 客户续费$12,000         → Rule: 增量归因           → 可归因收入$3,600按比例:
                       source: triageagent           已确认；增购$3,000         分流10%/技术支持40%       分流$360
技术支持Agent解决关键  → conversation (T2)                                      挽留30%/客户成功20%       技术支持$1,440
                       source: techsupportagent                                                              挽留$1,080
挽留Agent主动干预      → custom (T3)                                                                       客户成功$720
                       source: retentionagent
客户成功Agent发现增购  → crm (T5)
                       source: successagent
客户确认续费+增购      → payment (T7)
                       source: billing-system
```

### CTRS v1.2 验证

| 验证项 | 状态 | 说明 |
|--------|------|------|
| Claim → 多方声明 | ✅ 完全支撑 | 4 个 Agent + 客户作为 party |
| Evidence 覆盖 | ✅ 完全支撑 | `conversation`、`custom`、`crm`、`payment` 四种类型 |
| Rule 一等对象 | ✅ 完全支撑 | 增量归因规则（基于对照组续费率差） |
| Attribution 计算 | ✅ 完全支撑 | 但 `confidence` 可能较低（因果推断不确定性） |
| Settlement 分润 | ✅ 完全支撑 | $3,600 按 10/40/30/20 分配 |
| VR-101~VR-402 | ✅ 全部通过 | |

⚠️ **边缘情况**：客服归因的 `confidence` 天然较低（0.4–0.7 范围），因为"续费归因"涉及反事实推断。CTRS v1.2 的 `confidence` 字段（0-1 浮点数）可以表达这种不确定性，但消费方如何使用低置信度归因做结算决策，属于业务规则范畴，不是规范缺失。

### 一句话结论

这个场景证明 Fly 解决了 **客服从成本中心到利润贡献者的量化问题** ——没有信任链，客服 Agent 永远无法证明"我的服务让客户续费了"。

## 场景 4：AI 销售提成

### 场景标题

多角色销售协作提成：SDR Agent 挖掘线索 → AE Agent 签约，佣金怎么分？

### 参与者

| Agent | 角色 | 所属方 |
|-------|------|--------|
| SDR-Agent | 线索挖掘与预约 | 销售平台 A |
| ContentAgent | 内容营销（白皮书/案例） | 营销平台 B |
| SolutionAgent | 方案演示与定制 | 解决方案团队 C |
| AE-Agent | 商务谈判与签约 | 销售平台 A |

### 商业行为

SDR Agent 通过自动化外展触达潜在客户，内容 Agent 的白皮书建立了客户信任，解决方案 Agent 做了定制化 Demo 打消了技术疑虑，AE Agent 在最终谈判中签约——4 个 Agent 在 6–18 个月的销售周期中协作促成交易。

### 金额示例

| 项目 | 金额 |
|------|------|
| ACV（年合同价值） | $50,000 |
| 佣金率 | 10%（SaaS 行业中位数 11.5%） |
| 佣金池 | $5,000 |
| SDR Agent | $1,000（20%） |
| 内容 Agent | $750（15%） |
| 解决方案 Agent | $1,250（25%） |
| AE Agent | $2,000（40%） |

> 行业参考：SDR 底薪:浮动 = 70:30，按预约会议 $20–$200 或 SQL $100–$250 计酬；AE 佣金率 8%–12% ACV；Split Commission 中 SDR 获 AE 佣金的 15%–20%；82% 的 SaaS 公司使用加速器机制；70% 的 B2B 销售代表未达配额。

### 归因问题：为什么需要 Fly？

| 没有Fly | 后果 |
|---------|------|
| Inbound vs Outbound 归因模糊——客户自己下载白皮书后预约，SDR 只是接了电话 | SDR 按"会议数"计酬，不关心成交质量 |
| 6–18 个月销售周期中 SDR/AE 可能换了好几轮 | 谁该获得佣金无法追溯 |
| 解决方案 Agent 做了大量 Demo 但无独立佣金 | 前端 90% 工作由非 AE 完成，AE 拿走大部分佣金 |
| 内容 Agent 建立了信任但无法量化白皮书的转化贡献 | 内容创作无收入激励 |

### 完整链路

```
商业行为                Evidence                    Commercial Fact           Attribution               Settlement
─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
SDR Agent首次触达      → crm (T1)                  → ACV $50,000合同         → Rule: 多触点归因         → 佣金$5,000按比例:
                       source: sdr-agent             已签约                     SDR20%/内容15%           SDR$1,000
内容Agent白皮书下载    → webhook (T5)                                          解决方案25%/AE40%         内容$750
                       source: contentagent                                                                   解决方案$1,250
解决方案Agent做Demo    → conversation (T12)                                                                 AE$2,000
                       source: solutionagent
AE Agent谈判签约       → payment (T45)
                       source: ae-agent
```

### CTRS v1.2 验证

| 验证项 | 状态 | 说明 |
|--------|------|------|
| Claim → 多方声明 | ✅ 完全支撑 | 4 个 Agent 作为 party |
| Evidence 覆盖 | ✅ 完全支撑 | `crm`、`webhook`、`conversation`、`payment` 四种类型 |
| Rule 一等对象 | ✅ 完全支撑 | 多触点归因规则，时间跨度通过时间戳表达 |
| Attribution 计算 | ✅ 完全支撑 | 长周期证据链通过多条 Evidence 的时间戳自然形成 |
| Settlement 分润 | ✅ 完全支撑 | $5,000 按 20/15/25/40 分配 |
| VR-101~VR-402 | ✅ 全部通过 | |

✅ **完全支撑**。长销售周期的归因正是 CTRS 多 Evidence 链路的优势场景。

### 一句话结论

这个场景证明 Fly 解决了 **B2B 销售中多角色协作的佣金分配** 问题——没有信任链，SDR 和解决方案 Agent 的贡献永远被 AE 的"last-touch"掩盖。

## 场景 5：AI 创作分润

### 场景标题

内容创作-分发-变现链路分润：创作 Agent 写了 80%，分发 Agent 推广，钱怎么分？

### 参与者

| Agent | 角色 | 所属方 |
|-------|------|--------|
| ContentGenAgent | 内容生成 | 创作平台 A |
| EditOptAgent | 编辑优化（SEO/标题） | 优化平台 B |
| DistributionAgent | 多渠道分发 | 分发平台 C |
| MonetizeAgent | 商业变现 | 变现平台 D |
| HumanCreator | 人类创作者 | 个人 |

### 商业行为

内容生成 Agent 创作了 80% 的课程内容，编辑优化 Agent 将转化率提升了 30%，分发 Agent 在 10 个渠道投放推广，变现 Agent 匹配广告和带货选品，人类创作者修改和出镜——5 方协作产出一门知识付费课程。

### 金额示例

| 项目 | 金额 |
|------|------|
| 课程总收入 | $5,000 |
| 创作生成 Agent | $1,250（25%） |
| 编辑优化 Agent | $500（10%） |
| 分发推广 Agent | $1,500（30%） |
| 变现运营 Agent | $750（15%） |
| 人类创作者 | $1,000（20%） |

> 行业参考：YouTube 拿 45% 广告收入；头部 MCN 抽成 35%–45% 但服务价值系数仅 0.27；知识付费创作者分成 50%–70%；内容 vs 分发价值争议：好内容没分发 = 0 收入。

### 归因问题：为什么需要 Fly？

| 没有Fly | 后果 |

### 10/10 是否都能用 CTRS v1.2 走通？

**是的，10/10 场景的核心链路都能走通。**

每个场景的 "商业行为 → Evidence → Commercial Fact → Attribution → Settlement" 完整链路都可以用 CTRS v1.2 表达：

- **Claim → Evidence**：7 种 Evidence 类型（`webhook`、`conversation`、`mcp`、`payment`、`crm`、`stripe`、`custom`）覆盖了所有场景
- **Rule → Attribution**：Rule 作为一等对象 + Hash 绑定，归因规则不可篡改
- **Attribution → Settlement**：贡献百分比 + 分润金额，数学正确性可验证（VR-401）

### 哪些场景需要 v1.2 以上的增强？

| 增强级别 | 涉及场景 | 增强内容 |
|---------|---------|---------|
| **Governance Extension** | 场景 2、8、9 | Rule/Issuer 注册表验证，跨平台信任建立 |
| **Enterprise Extension** | 场景 2、5、10 | 多 Settlement 分期结算、审计追踪、争议工作流 |
| **v1.3 可选扩展** | 场景 1、3、7 | Evidence `issuer` 逐条信任验证、归因窗口标准化、`confidence` 语义细化 |

**关键发现**：没有一个场景因为 CTRS v1.2 的结构性缺陷而无法走通。所有"限制"都属于：
1. 边缘情况（归因窗口、长尾收入）→ 可通过 Rule parameters 扩展
2. 低置信度归因（客服、采购的"避免成本"）→ CTRS 的 `confidence` 字段已可表达，业务方自行决策
3. 企业级需求（审计追踪、分期结算）→ Enterprise Extension 已有路线图

### 核心结论：CTRS v1.2 是否足够？

**CTRS v1.2 足够支撑所有 10 个场景的核心链路。**

更准确地说：

1. **Core 级别（VR-101~VR-402）足以验证所有 10 个场景的结构完整性和数学正确性**
2. **3 个场景需要 Governance Extension 的注册表验证来建立跨平台信任**
3. **3 个场景需要 Enterprise Extension 的分期结算和审计追踪来满足企业合规需求**
4. **v1.3 的最小扩展（Evidence `issuer` 字段、归因窗口标准化）可以让低置信度场景更健壮，但不是必须的**

CTRS v1.2 的 Compliance Levels 设计（Core → Governance → Enterprise）恰好匹配了这个分层需求——实现者可以从 Core 起步，按需升级。

### 冻结原则验证

基于 10 个场景的验证，我们检查是否有规范修改的必要：

| 潜在修改 | 需要的场景数 | 是否触发冻结原则 | 结论 |

### 为什么 Fly 是必需的

10 个场景共同证明了一个事实：**Agent Economy 中没有任何参与方拥有完整事实**。

| 场景 | 谁看不到全链路？ |
|------|---------------|
| AI 导购 | 推荐平台看到推荐，支付平台看到付款，没人看到全链路 |
| AI 招聘 | AI 平台看到筛选，猎头看到推荐，没人知道谁先发现了候选人 |
| AI 客服 | 客服知道解决问题，财务知道续费，但两者的因果关系无法证明 |
| AI 销售 | SDR 知道首次触达，AE 知道签约，6 个月中间链路是黑箱 |
| AI 创作 | 创作者知道内容，分发平台知道流量，没人知道内容值多少钱 |
| AI 搜索 | 搜索引擎看到查询，对比工具看到决策，没人知道信息值多少 |
| AI 采购 | 比价工具看到价差，谈判 Agent 看到折扣，没人知道谁贡献了节省 |
| AI SaaS | 集成商知道对接，运维知道维护，但续费归因无依据 |

Fly 解决的不是数据格式问题，而是 **多方互不信任环境下的商业事实构建问题**。CTRS 的价值在于：让每一份商业贡献都有证据支撑、规则可查、机器可验证——不是"我觉得"，而是"Fly 证明"。
